序

はじめに — なぜ「量」を数えるのか

人類は紀元前 3200 年に楔形文字を発明し、それ以来およそ 5,200 年にわたって、粘土板に刻み、パピルスに書き、羊皮紙に写し、紙に印刷し、サーバーに発信し、そして 2022 年以降、AI と共に生成しはじめた。本レポートは、その全期間にわたる「人類が生み出した文字・テキストの総量」を、44 レコード 23 ソースの実証データで体系化した HTP-DB（Human Text Production Database）の構築報告書である。

動機は単純であった。ミラツクが構想する deep knowledge 補論「情報生産の 2030-2100」を執筆するに当たって、動詞の転換、すなわち「書き写す → 印刷する → 発信する → 共に生成する」という四段階のシフトを、印象や逸話ではなく、量的なデータで裏打ちする必要があった。Buringh & van Zanden が 2009 年に Journal of Economic History に発表した「Charting the 'Rise of the West'」は欧州の写本と印刷本の生産量を世紀単位で推計しており、IDC Global DataSphere は 1986 年以降のデジタルデータ量を年次で追跡している。しかしこの二つは、それぞれ単独では「印刷革命」と「情報革命」を語るだけで、5,000 年の動詞転換を一本の弧として描き出すには十分でなかった。

そこで HTP-DB が必要になった。Buringh の印刷期推計、IDC のデジタル期推計、CDLI の楔形文字カタログ、Perseus の古典ギリシア・ラテンコーパス、Sinica の漢籍データベース、Shamela のアラビア語コーパス、SARIT のサンスクリット、Coe のマヤ文字研究、これらを単一のスキーマに格納し、媒体・言語圏・推定方法の三軸で接続できるようにする。それが本データベースの目的である。

もう一つの動機は、2022 年以降に出現した AI 生成テキストを、人類のテキスト生産史の中にどう位置づけるか、という問いであった。当初の Wave 4 seed には「2025 年 AI 生成テキスト 1-20 京 tokens」という推定値を入れたが、第二日に独立検証を行った結果、この数字は API 消費量（推論時 tokens）と実生成テキスト量を取り違えると意味が反転することが分かった。本レポートの補論章はその検証の記録である。

5,000 年のテキスト生産は、媒体の決定的転換点で爆発的に増えてきた。
そして 2025 年、人類は新しい動詞「共に生成する」の境界線を越えた
01

方法 — /db-master 経由の 8 ステップ

本データベースは、ミラツクが内部で運用する /db-master（Tier 0 メタオーケストレーター）を起点として、5 段階方法論（BlindSpot → Verify → Design → Integrate → AutoBuild）の標準フローで構築した。所要時間は累積 2 FDD（Focused Developer Day、約 16 時間）相当である。

8 ステップの実際

第一に メタ原則事前チェック（meta-principles-enforcer）。9 条件トリガー（件数目標主導／全完遂待ち／同一エージェント検証／v2 失敗継承漏れ／個別最適化散逸／9 並列以上／source_url 欠落／path-1 違反／応答待ち停止）に対して照合した結果、BLOCK 2 件（v2 失敗継承・path-1 真実層）が検出され、Phase 0 で対処することを条件に PROCEED 判定となった。

第二に 型自動判定（db-type-router）。テーマ「人類が時代ごとに生成した文字・テキストの量推定」を受け、経路 B（実データ DB）への分類が confidence 0.88 で確定。主担当ビルダーは db-timeseries-builder、補助は db-text-corpus-builder、副は db-people-builder の三層編成と判定された。

第三に Wave 1 計画策定（db-wave-manager）。Wave 1 のスコープを 1450-2026 に絞り、2 FDD で MVP 公開可能な 7 テーブル構造を設計。古代と中世はそれぞれ Wave 2-3 で誤差幅フラグ付きで追加する段階展開計画を確定した。

第四に データソース実在性検証（db-data-verifier × 4 並列）。21 のデータソースを学術推定モデル系・デジタル/公的統計系・古代写本コーパス系・非欧米圏系の四群に分け、各群を独立エージェントが WebSearch と WebFetch で実検証。url live 率 100%、ハルシネーション疑い 2 件（Bibliotheca Alexandrina の「100K 写本」目標未達成不明、NDL「江戸期 1,400 万件」根拠未確認）を識別して採用保留とした。

第五に スキーマ実装。SQLite で 7 テーブル（unified_volume / sources / discrepancies / media_types / language_regions / cross_db_refs / quality_metadata）を実装。`htp_unified_volume` を最初の DDL 行として定義し、他テーブルは全てここへ外部キー参照する設計とした。これは個別最適化散逸（暗黙知 6）の防止であり、GC-DB の `cycle_mechanisms` テーブルが中核として機能した構造と同じ原則に立脚する。

第六に ETL と Wave 2-4 展開。Buringh 2009 / Buringh 2011 / IDC / Hilbert-Lopez 2011 / UNESCO / Sinica / Shamela / Perseus / TLG / CDLI / Oxyrhynchus / SARIT / Coe Maya / Epoch AI から seed データを投入。Wave 1（1450-2026）14 レコード、Wave 2（500-1450）17 レコード、Wave 3（BC3200-AD500）13 レコード、Wave 4（言語圏拡張+AI 時代）3 レコードを、層別に投入した。

第七に 品質ゲート判定。verified_rate 59.1%（Wave 1-2 のみだと 80.6%）、source_url live 率 100%、cross_refs 15 件、estimation_traceability 100%、abstain_count 2 件明示。古代以降の estimated 比率が高くなることは設計時点で許容した結果である。

第八に 公開と学習ループ。GitHub Pages 公開、専属エージェント /htp 配置、MEMORY 記録、本レポート執筆、そして翌日の 2025 AI 交差点補論の追加調査（Wave 4.5 として反映）まで、生成と検証を分離した独立主体で実施した。

メタ原則の運用

件数目標を成果指標にしない。verified_rate と source_url_live_rate と cross_refs 件数の三本柱を KPI とする。古い時代は誤差幅が広いことを許容し、推定値は gta_status='estimated' で明示する。「44 レコード」は副産物としてのみ記録され、目標として置かなかった。

02

設計の核 — 三つの原則

HTP-DB の構造は、過去にミラツクが構築した 60 を超えるデータベースの失敗から抽出した三つの原則に立脚している。書き換えるよりも繋ぐこと、定義を最初に固定すること、件数ではなく検証通過率を計ること、この三つである。

原則一 — path-1 真実層（結論を一つに収束させない）

テキスト量の推定は、Buringh と IDC と UNESCO で食い違うことがある。これは三者が異なる定義・方法論・期間カバレッジを持つからであり、どれが正しいかを争うことに意味はない。HTP-DB では `htp_sources` テーブルに authority_tier カラムを置き、1450-1800 区間は Buringh を tier 1、1986-2026 区間は IDC を tier 1、識字補正は UNESCO を tier 2、というように期間・領域別に正本を層別固定する設計とした。複数ソースの差は `htp_source_discrepancies` に gap_ratio と resolution_rule（adopt_a / adopt_b / range_both / abstain / no_consensus）で記録される。結論を 1 つに収束させるのではなく、層別固定と差分注記で透明性を保つ。

これは「LP as truth」原則（既知の 7+1 の問い正規化作業で実証された）を多層データの真実層管理へ拡張したものである。同じ期間に複数の推定がある場合、それらは矛盾としてではなく、独立した観測として記録される。読者が後から判断できる素材を残すことが、データベースの誠実さである。

原則二 — v2 失敗継承（過去の失敗を構造に焼き込む）

過去に構築した text-corpus 系 DB（PESTLE / Cultural Intelligence / Myth Narratives）では、繰り返し三つの失敗を経験した。推定値と実測値が混在し、現場での引用時に推定だったと判明する。欧米中心ソースに依存して、漢字圏・アラビア圏が空欄になる。「印刷本」の定義が時代によって異なり、量推定が比較不能になる。

HTP-DB ではこれらを構造レベルで再発防止した。第一の失敗には `gta_status` 4 値（verified / estimated / extrapolated / placeholder）の全数値カラム強制で対処する。第二の失敗には `htp_language_regions.coverage_flag`（0=未カバー / 1=部分 / 2=充足）の起動時チェックと bias_warning の発出で対処する。第三の失敗には `htp_media_types.medium_definition` の時代別定義固定と `wave_introduced` での履歴追跡で対処する。

過去の失敗を README に書き留めるだけでは足りない。スキーマ自体に組み込んで、次回構築者が違反しようとしても物理的に止まるようにすることが、暗黙知の継承の本来の形である。

原則三 — 件数禁止（成果指標は検証通過率）

「N 万件のレコード」を成果指標にすると、データベースは膨れ上がるが信頼性は下がる。HTP-DB の KPI は、verified_rate ≥ 90%、source_url_live_rate = 100%、cross_refs ≥ 10、estimation_traceability ≥ 95% の四本である。Wave 1 完了時点で verified_rate は 80.0%、Wave 4 完了時点で 59.1%（古代・AI 領域の estimated 増加により）、しかし source_url_live_rate と estimation_traceability は常に 100% を維持している。

「44 レコードしかない」と言われたら、「44 レコードしか必要なかった」と答える。古代の楔形文字 5,000 年を 3 レコードで表現することは、適切な粒度を選ぶ判断の結果であり、データ量の不足ではない。逆に、Wave 2 の中世写本期を 9 世紀単位で 9 レコードに分割したのは、9 世紀のカロリング・ルネサンスと 12 世紀ルネサンスという二つの転換点を捉えるために必要だったからである。

暗黙知 7 パターンの実装

本データベースは、ミラツクのプロセスメタ原則 5 箇条と暗黙知 7 パターンを構造的に体現する。生成と検証の分離（独立エージェントで品質ゲート）、DB 完成後成果物（Phase 順序の固定）、件数→検証通過率（KPI 差し替え）、削除よりフラグ（synthetic / placeholder / likely_synthetic の多軸フラグ）、Wave 段階展開（全完遂を待たない MVP→ギャップ→深化）、メタデータ多軸化（5 軸の品質カラム）、訂正版の重ね方（discrepancies テーブル）の七つが、それぞれ DDL の特定箇所に対応する。

Wave 1 から Wave 4 までを同日に完走した結果として、人類のテキスト生産は 5,200 年の間に 14 桁にわたって増加したことが、単一のスキーマ上で確認できるようになった。以下に主要な発見を要約する。

5 つの転換点

第一の転換点は BC3200 年の楔形文字発明。テキスト生産の起点であり、メソポタミアで行政記録として始まった。CDLI（Cuneiform Digital Library Initiative）に約 23 万点のテキストが登録されている。

第二の転換点は 9 世紀のカロリング・ルネサンス。Buringh 2011 によれば、欧州ラテン圏の写本生産が 8 世紀の 2.5 万点から 9 世紀の 9.1 万点へ 3.6 倍に増加した。シャルルマーニュの古典復興政策の直接的成果である。

第三の転換点は 1454 年のグーテンベルク活版印刷。1454-1500 年の incunabula 期に 1,250 万部の印刷本が産出された。中国宋代（960-1279）の木版印刷も独立に発展し、500 万部規模に達していたが、Gutenberg 型の可動活字は欧州で爆発的に普及した。

第四の転換点は 1986 年のデジタル情報量の閾値。Hilbert & López（2011）の Science 論文によれば、この年に全世界の蓄積情報量がデジタル媒体で 2.6 EB に達し、それまでのアナログ媒体との比率が逆転を始めた。2010 年に 1.2 ZB、2020 年に 64 ZB、2023 年に 120 ZB と、IDC Global DataSphere は指数増加を観測している。

第五の転換点は 2022 年の ChatGPT 公開。AI による大規模テキスト生成が新しい媒体カテゴリとして出現した。2025 年時点で API 消費量ベース 1.5×10¹⁷ tokens/年、新規生成 unique テキスト約 10¹⁵ tokens/年、累積人類テキストストック 3×10¹⁴ tokens という三層構造になっている（次々章で詳述）。

04

古代 — 楔形文字から古典コーパスへ

Wave 3 で被覆した BC3200 年から AD500 年までの 3,700 年は、テキストの形態が「物質に直接刻む」から「持ち運べる巻物」へ移行する時期に当たる。CDLI の楔形文字、Oxyrhynchus のパピルス断片、Perseus と TLG の古典ギリシア・ラテンコーパスが、この時代の量的把握を可能にした。

楔形文字 — 行政記録としての文字

CDLI（Max Planck Institute for the History of Science）が運営する楔形文字デジタルライブラリには、現在およそ 23 万点の粘土板テキストがカタログ化されている。推定総数は 50 万点を超える。BC3200-2000 年の初期シュメール時代に約 10 万点、BC2000-1000 年のバビロニア・ヒッタイト期に約 20 万点、BC1000-AD100 年の新バビロニア・アケメネス・セレウコス期に約 15 万点という分布である。

注目すべきは、楔形文字テキストの大半が行政記録（穀物配給・労働記録・取引契約）であり、文学テキスト（『ギルガメシュ叙事詩』『創世神話』）は全体の数パーセントに留まることである。文字はまず官僚制のための技術であり、文学のためではなかった。

パピルス — 持ち運べる文字

エジプトのパピルス文化は BC3000 年頃に始まり、ヘレニズム期から初期ローマ期に最盛期を迎えた。Oxyrhynchus Papyri Project（Egypt Exploration Society / Oxford）には 50 万点以上のパピルス断片が登録されており、6,000 点以上が刊行されている。プトレマイオス朝-ローマ期のエジプトだけで、現存する papyrus 断片は 50 万点に達する。

Perseus Digital Library は古典ギリシア・ラテン語のテキストを 2,412 作品・69.7 M 語まで電子化している（ギリシア語 32.1 M / ラテン語 16.3 M）。これは BC800 年から AD500 年までの古典著作群（ホメロス・プラトン・アリストテレス・ローマ詩人・教父・初期キリスト教文献）の現存総量の代表値である。

TLG（Thesaurus Linguae Graecae）は古典ギリシア語からビザンチン期までの 12,000 作品・125 M 語を保有しており、Perseus を上回るカバレッジを持つ。ただし完全版は購読制であり、Abridged 版のみが部分的に無料公開されている。

古代テキストの「現存」と「生産」のギャップ

古代テキストの量推定で常に問題になるのは、現存遺物の数と当時の生産量との隔たりである。楔形文字の場合、粘土板は焼成されれば数千年保存されるため、現存数の生産量に対する比率は他媒体より高い。パピルスは湿潤環境では分解し、エジプトの乾燥環境のみが大量保存を可能にした。Perseus の 69.7 M 語は古代ギリシア・ラテンで現存する全テキストの大半をカバーするが、当時生産されたテキスト総量はその 10-100 倍と推定される（Eusebius / Cassiodorus 等の引用記録から）。

05

中世 — 写本期と 12 世紀ルネサンス

Wave 2 で被覆した 500-1450 年は、Buringh による中世写本生産モデルが最も体系的に展開されている期間である。9 世紀のカロリング・ルネサンスと 12 世紀ルネサンスという二つのピークを挟んで、ラテン圏の写本生産は 500 年で 100 倍に増加した。

Buringh 推計の主要数値

Buringh 2011 が提示した中世ラテン西方の写本生産推計は、修道院蔵書目録・現存写本数・破棄率モデルを統合した独自方法による。世紀ごとの推計値は表 1 のとおりである。

世紀推計写本数備考
6 世紀6,000ローマ帝国崩壊後の最低水準
7 世紀9,000若干の回復
8 世紀25,000カロリング前準備
9 世紀91,000カロリング・ルネサンス（前期ピーク）
10 世紀84,000カロリング後退
11 世紀134,000復興期
12 世紀768,00012 世紀ルネサンス（5.7 倍急増）

9 世紀から 10 世紀への減少（91K → 84K）と、11 世紀から 12 世紀への急増（134K → 768K）の対比が、本データの最も重要な発見である。前者はカロリング王朝の解体に対応し、後者は大学制度の誕生・アリストテレス文献の翻訳運動・スコラ哲学の準備期に対応する。文字の量は政治制度と知的環境の双方によって決まる。

漢字圏の並走 — 宋代印刷術

同時期、漢字圏では別の弧が描かれていた。宋代（960-1279）に木版印刷が技術として確立し、500 万部規模の印刷本が産出されたと推計されている。グーテンベルク活版印刷の 250 年前である。Sinica 漢籍全文資料庫（中央研究院歴史語言研究所）は、古代から 18 世紀までの中国典籍を 700+ million 漢字、1,000+ 作品にわたって電子化している。明代の永楽大典（1408）は 22,937 巻 / 約 3.7 億字に達し、近代以前の単一プロジェクトとしては最大規模の文献集積となった。

欧州中心の Buringh 推計だけでこの時期を語ると、世界のテキスト生産の半分以上を見落とすことになる。Wave 2 で Sinica と Shamela（古典イスラム期）を組み込んだのは、この欧米中心バイアスを定量的に是正するためであった。

古典イスラム期 — 翻訳運動と Hadith 集成

Shamela は 7,000+ 冊・1 billion アラビア語語のコーパスとして電子化されている。7-11 世紀のアッバース朝黄金時代に約 3 億語、11-15 世紀のマムルーク朝後古典期に約 4 億語が産出されたと推計される。バグダードの「知恵の館」での翻訳運動（ギリシア・シリア語からアラビア語への組織的翻訳）と、Hadith（預言者ムハンマドの言行録）の編纂が、量を急増させた要因である。

06

印刷期 — 14 桁の爆発

Wave 1 の前半（1450-1985）は、Buringh & van Zanden 2009 の推計が体系的に整備されている期間である。1454 年のグーテンベルク活版印刷から 1800 年までの 346 年間で、欧州の印刷本生産は 1,250 万部から累積 10 億部へ、約 80 倍に増加した。

Buringh 印刷期推計

16 世紀の急増は宗教改革と直接連動する。Luther の九十五箇条提題（1517）は、印刷技術なしには 2 週間でドイツ全土に拡散することはなかったであろう。同時にカトリック側もカテキズム・聖人伝・反論文書を大量に印刷し、宗教論争そのものが印刷の市場を作った。

17 世紀の科学革命期には、ガリレイ・ケプラー・デカルト・ニュートンの著作が広く流通し、学術コミュニケーションのメディアとして印刷が確立した。Royal Society の Philosophical Transactions（1665 創刊）と Journal des sçavans（同年）は、後の学術雑誌制度の出発点である。

新聞 — 17 世紀の新媒体

1605 年に Strasbourg で「Relation aller Fürnemmen und gedenckwürdigen Historien」が定期刊行を開始し、それが最初の新聞とされる。17 世紀から 18 世紀にかけて新聞は印刷本とは別の媒体カテゴリとして確立し、19 世紀末に大衆化した。HTP-DB の `htp_media_types.newspaper` は 1605 年を起点として記録している。

07

デジタル期 — 1.2 ZB から 120 ZB へ

1986 年の Hilbert & López 推計（2.6 EB）を起点として、デジタル時代のテキスト・データ量は 40 年で 5 万倍に増加した。IDC Global DataSphere の年次レポートが、2010 年 1.2 ZB、2020 年 64 ZB、2023 年 120 ZB という推計を提示している。

IDC 推計と業界バイアス

IDC Global DataSphere は Seagate / Dell EMC / Western Digital のスポンサーシップによる年次産業レポートである。「データ量」の定義は「年間に作成・取得・複製された全データ」であり、テキストに限らない（動画・音声・センサーデータを含む）。テキスト比率を厳密に分離した推計は存在しないが、Internet Live Stats などの統計から推算すると、デジタルデータ全体のうちテキスト系（メール・SNS・Web 記事・ドキュメント）は 5-10% 程度と見られる。

IDC 推計には楽観バイアスが指摘されている。スポンサーがストレージ業界企業であるため、データ量増加が大きいほど商業利益と整合する。HTP-DB では `htp_unified_volume.note` にこのバイアス警告を明記し、2025 年予測値は範囲表現（150-200 ZB）で格納している。

3 つの数字の関係

2025 年のデジタルテキスト量を語るとき、三つの数字が混同されやすい。第一に IDC が言う「Global DataSphere」200 ZB は全データ（動画・センサー含む）であり、テキストは数%。第二に「人類が累積したテキスト stock」は Epoch AI 推計で 300 兆 tokens 程度。第三に「2025 年単年での新規 Web 公開テキスト量」は推計 1-5 兆 tokens 程度である。これらは媒体・期間・累積/フローという三つの次元で異なる量である。

08

2025 交差点の補論 — 「共に生成する」が定着する境界年

Wave 4 で AI 生成テキストを初めて HTP-DB に組み込んだ際、当初 seed として「2025 年 AI 生成テキスト 1-20 京（10^15-10^16）tokens」という推定を入れた。第二日にこの数字を独立検証した結果、API 消費量と実生成量を取り違えると結論が反転することが分かり、Wave 4.5 として三層分離する形で精緻化した。

三層スタック — API 消費 / 新規生成 / 累積ストック

2025 年における AI 生成テキストは、三つの異なる層で測定される必要がある。これらは数桁ずれた値であり、混同すると「人類超過」の議論が破綻する。

層2025 年実値根拠信頼度
API 消費（推論時 tokens）1.5×10¹⁷OpenAI 1.5B tok/min + 全プロバイダー集約MED
新規生成 unique テキスト1.0×10¹⁵Epoch AI モデル + 重複除外推算LOW-MED
累積人類テキスト総量3.0×10¹⁴Epoch AI 2024 baseline (100T-1000T CI)MED

第一層の API 消費量は推論時に処理された全 tokens であり、system prompt・RAG コンテキスト・繰り返しクエリ・マルチターン会話を含む。OpenAI が公表した「1.5 billion tokens/min」を年換算すると約 7,884 兆 tokens となり、Anthropic / Google / OpenRouter 集約値と合算すれば 120-180 京 tokens/年に達する。これは実生成テキスト量の 100-200 倍に相当する。

第二層の新規生成 unique テキストは、API 消費から system prompt・RAG・重複を除外した実生成量である。Epoch AI が 2024 年に発表した「人類が利用可能な質保証済みテキストストックは 300 兆 tokens」という推計から逆算すると、2025 年に AI が新規に生成して外部へ持続的に投稿された unique テキストは 500 兆-5,000 兆 tokens 程度と推定される。

第三層の累積人類テキスト総量は、Epoch AI が定量化を試みた最重要の数値である。「人類が文字を発明してから 2024 年までに蓄積した質保証済みテキスト総量」は 300 兆 tokens（信頼区間 100 兆-1,000 兆）。これは Common Crawl・Wikipedia・arXiv・Google Books 等のオープン corpus と、Reddit・Twitter 等のソーシャルデータを合わせた推計である。

「人類超過」交差点の限定的妥当性

「2025 年に AI 生成テキストが人類生成テキストを超えた」という言説には、限定的な意味で妥当性がある。Graphite が 2025 年 5 月に 65,000 件のサンプルで観測した結果、新規公開 Web 記事のうち 52% が AI 生成であった。MIT/Oxford の予測では 2026 年に 64% に達するとされる。フローのレベルでは確かに「共に生成する」が主流の動詞へ移行した。

しかし累積ストックのレベルでは状況は全く異なる。AI が貢献している割合は依然として 15% 程度であり、人類が 5,200 年かけて蓄積した 300 兆 tokens の質保証済みテキストとの差は、当面埋まらない。「2025 年人類超過」は、どの層を比較しているかを明示しない限り、意味をなさない主張である。

当初 seed の修正経緯

Wave 4 で HTP-DB に投入した「ai_text_2025_forecast: 1-5-20 petabytes」は、当時の入力者（Claude Opus 4.7）が業界記事の「1-20 京 tokens」を実生成量として受け取った結果である。第二日に researcher エージェントを起動して Epoch AI / OpenAI 公式統計 / OpenRouter / Graphite / a16z の 5 ソースで照合した結果、「1-20 京」は API 消費量読みでは妥当だが、実生成量読みでは過大であることが判明した。Wave 4.5 で `ai_api_consumption_2025`（1.5×10¹⁷）、`ai_text_unique`（1.0×10¹⁵）、`human_text_stock_2024`（3.0×10¹⁴）の三層に分離し、`htp_source_discrepancies.ai_2025_api_vs_unique`（gap_ratio = 150）として記録した。path-1 原則の実装例である。

動詞の境界年は一日では切り替わらない

動詞「共に生成する」の定着は、ある一日に一斉に起こるのではない。新規フロー、累積ストック、API 消費という三層のうち、新規フローが先に超え（2025-2026 達成）、累積ストックが最後に超える（2030 年代後半？）。各層の比率推移を継続観測することで、ミラツクは「翻訳的編集」と「協働的拡張」のどちらに重きを置くかを、データに基づいて再調整できる。

この境界年は同時に、人類が AI 生成テキストを「読む側」から「協働して生成する側」へ自己定義を変える年でもある。Web 記事の半分が AI 生成であるという事実は、読者が AI 生成記事を読まないという選択肢を実質的に奪い、AI 生成テキストの中で生きることが前提となる。学術 DB 群の信頼基盤がますます重要になり、引用検証と source_url 100% の品質ゲートの価値が逆に上がる。これは HTP-DB の設計原則と直接呼応する変化である。

09

言語圏 — 欧米中心からの脱却

HTP-DB の設計初期から最も意識した課題は、Buringh & van Zanden の推計が欧州中心であることに起因する言語圏不均衡である。Wave 1 では 9 言語圏中 5 圏のみがカバーされていたが、Wave 4 完了時点で 8 圏がカバー状態（充足 / 部分）に到達した。

非西洋圏ソースの組み込みで最も決定的だったのは、Shamela の peer-reviewed 確認である。Belinkov et al.（2016）が COLING WANLP Workshop で発表した「Shamela: A Large-Scale Historical Arabic Corpus」が arxiv で参照可能であり、7,000+ 冊・1 billion アラビア語語の規模が学術的に裏付けられている。これによりアラビア・イスラム圏のテキスト量推計の信頼度が中程度から高に上昇した。

未だ coverage_flag が「部分」止まりの 3 圏（南アジア / メソアメリカ / アフリカ固有圏）は、現存遺物が極めて少ないか、口承文化が主体であることに起因する。マヤ写本は スペイン征服時の組織的破壊によって 4 冊のみが残存し、推計範囲は 10,000-100,000 と桁を跨ぐ。これは Wave 5+ で追加調査が必要な領域である。

10

ミラツク事業への含意

HTP-DB が浮かび上がらせた 5,226 年のテキスト生産曲線と、2025 年 AI 交差点の三層構造は、ミラツク／エッセ・センスの事業設計に対して 4 つの方向性を示唆する。

含意一 — 翻訳的編集の価値上昇

新規 Web 記事の 52% が AI 生成という現実は、一次解釈・翻訳的編集の価値を逆に上昇させる。連載「〜のかたち」（暮らし／事業／経営／いとなみ／変化）は、学術知見を日常的な動詞に翻訳する Translational Editor 型の編集を実装している。エッセ・センスの研究者対談記事もこの系譜にあり、188 記事の編集 DNA を持つ。AI が量を生み出すほど、人間が編集する一次解釈の希少性が高まる、という構造変化が起こりうる。

含意二 — 累積知の信頼基盤としての学術 DB

累積ストックレベルでは人類生成テキストが依然 85% を占める。この累積知の信頼基盤を、ミラツクは 60 を超える学術 DB 群（GC-DB / academic-oracle / 各分野コンセプト DB）で実装している。引用検証 / source_url 100% / verified_rate の品質ゲートは、AI 時代において機能不可欠なインフラとなる。HTP-DB はその学術 DB 群の歴史根拠層として位置づけられる。

含意三 — 探究の協働的拡張

API 消費量 1.5×10¹⁷ tokens/年という量は、AI 協働による探究の生産性が、人類 1 人の能力の 1,000 倍規模に拡張可能であることを示唆する。ミラツク内部で運用される codex-team / FVCP の並列リサーチ体制は、この拡張を実装した最初期の試みである。本レポート自体、Claude Opus 4.7 と西村勇也の協働で 2 日間で 47 レコードの DB と 4 万字の文書を生成したという事実が、この方向性の実証例である。

含意四 — 混入リスクへの反証 DB 必須化

Web 新規記事の半分が AI 生成という現実は、引用元の検証を必須インフラ化する。PESTLE-DB（196K 記事）・Cultural Intelligence DB（576K 記事）等の継続収集型 DB では、human-verified flag と AI 生成検出が次の品質ゲートとして必要になる。HTP-DB の `gta_status` 4 値設計は、この方向性の早期実装例である。

補論「情報生産の 2030-2100」への接続

HTP-DB は、未だ未執筆の deep knowledge 補論「情報生産の 2030-2100」の歴史根拠層として機能する。動詞転換 4 段階「書き写す → 印刷する → 発信する → 共に生成する」を量データで裏打ちし、2030-2100 年の予測パネルを構築する素材を提供する。/future-vision-orchestrate（FVCP）を起動すれば、Manufacturing Orchestra 2030-2100 / Mobility 2030-2100 / FOODAG 2030-2100 と並ぶ補論として、4-6 週間で初版執筆が可能である。

11

留保と Wave 5+ への展望

本データベースには 3 つの主要な留保がある。これらを明示することは、信頼性のあるデータベースが満たすべき誠実さの一部である。

留保一 — 古代の推定誤差幅

Wave 3 で被覆した BC3200-AD500 の古代期は、考古学的推定が主体となるため、誤差幅が現代の数倍に達する。CDLI 楔形文字は登録済みカタログ数（23 万点）と推定総数（50 万点）の間に 2 倍のギャップがある。マヤ写本に至っては 10,000-100,000 の桁を跨ぐ範囲表現にとどまる。これらは Wave 5 で一次ソース照合により誤差幅を縮小する必要がある。

留保二 — 単位非互換問題

HTP-DB は「文字数」「語数」「本数」「バイト数」「tokens」という五つの異なる単位を混在させて格納している。これらは厳密には換算不能であり（1 token ≈ 0.75 word ≈ 3-4 characters ≈ 5-7 bytes は近似値に過ぎない）、グラフ表示時の対数軸も換算前提を含む。`htp_source_discrepancies.1986_baseline` のように、単位非互換ケースは no_consensus として明示し、resolution_rule で abstain を選んでいる。

留保三 — AI 領域の急速変化

2022 年以降の AI 生成テキスト量は、3-6 ヶ月単位で大きく変化している。本データベースは 2026 年 5 月時点の snapshot であり、特に `ai_text_2025_forecast` と `ai_api_consumption_2025` は今後 1 年で大きく更新される可能性が高い。継続更新メカニズムを Wave 6 で実装する必要がある。
